# Research: IDS Refactor Foundation

## Decision 1: Build a provider-agnostic ingestion boundary first

**Decision**: Introduce a normalized incident ingestion boundary before tightly
embedding any one external detection engine into the backend request path.

**Rationale**: The current repository already has a local IDS middleware and a
mixed demo pipeline. A provider boundary lets the project ingest outputs from
mature external sources incrementally without committing to a large runtime
rewrite in the first refactor slice.

**Alternatives considered**:

- Continue expanding the current hand-written matcher. Rejected because it
  doubles down on the existing demo-heavy design and weakens detector trust.
- Hard-embed one new external detector immediately. Rejected because it raises
  delivery risk and increases merge conflicts before provenance and event-origin
  handling are solved.

## Decision 2: Separate event origin from detection source

**Decision**: Track `event origin` and `detection source` as separate concepts.

**Rationale**: A real event generated by a local rule, a demo event generated by
  a seed endpoint, and a real event generated by an upstream detector are not
  the same thing. The UI and metrics cannot stay trustworthy unless origin and
  source are independently visible.

**Alternatives considered**:

- Use a single attack-type or action field to imply origin. Rejected because it
  makes filtering brittle and encourages hidden demo contamination.

## Decision 3: Keep the current regex engine only as a transitional provider

**Decision**: Freeze the local request matcher as a transitional provider rather
than investing in broad new rule growth.

**Rationale**: The application still needs continuity while the refactor is
  underway, but future detection quality should come from mature sources or
  explicitly documented campus-specific custom logic.

**Alternatives considered**:

- Delete the local matcher immediately. Rejected because it would remove current
  visibility before replacement ingestion paths exist.
- Promote the local matcher as the primary long-term detector. Rejected because
  it does not meet the team's stated quality goals.

## Decision 4: Use a staged mature-source roadmap

**Decision**: Stage mature detector adoption in this order:

1. Normalize source metadata and ingestion contracts.
2. Support upstream web-attack detector outputs through the new ingestion path.
3. Support network-alert ingestion from EVE-style feeds.
4. Support quarantine and file-detection ingestion.
5. Support host-log detections through normalized log-rule mapping.

**Rationale**: This order improves trust and architecture immediately without
  forcing all detector families into the first branch slice.

**Alternatives considered**:

- Attempt a full detector-family migration in one phase. Rejected because the
  team still has conflict pressure and daily review requirements.

## Decision 5: Real metrics exclude demo/test data by default

**Decision**: All default IDS counts, charts, and unresolved queues represent
real incidents only; demo/test data is opt-in.

**Rationale**: This is the fastest way to restore reviewer trust in the
security-center outputs.

**Alternatives considered**:

- Keep mixed metrics with a visual badge. Rejected because the default view
  would remain misleading.

## Decision 6: Daily changelog and push are part of the refactor workflow

**Decision**: Treat `docs/ids-changelog.md` updates and daily branch pushes as
mandatory operational artifacts for this feature.

**Rationale**: The refactor is collaborative and will be reviewed incrementally.
Without daily traceability, the team will struggle to distinguish planned
architecture work from partial experiments.

**Alternatives considered**:

- Capture progress only in commit history. Rejected because the repository also
  contains unrelated domains and reviewers need a stable narrative of IDS work.
